ThousandEyes Cisco Router Enablement Guide

A step-by-step workflow for Cisco SEs to deploy ThousandEyes Enterprise Agents on Cisco routers via CLI or SD-WAN Manager, configure Traffic Insights forwarding, and set up comprehensive branch-to-cloud monitoring.

Cisco routers running IOS-XE (Catalyst 8100/8200/8300/8400/8500 series, ISR 1100X/4000, ASR 1000) support native ThousandEyes Enterprise Agent deployment through the Application Hosting framework. When combined with Cisco SD-WAN, this enables deep visibility into both overlay and underlay network performance from every branch site.

What This Guide Covers

  • Enterprise Agent Deployment on Cisco routers via CLI or SD-WAN Manager workflows
  • Traffic Insights Configuration using the router as a traffic forwarder
  • Recommended Test Configurations for branch-to-data-center and branch-to-cloud monitoring
  • SD-WAN Overlay & Underlay Monitoring for full-stack WAN visibility

🔎
Router vs. Switch Deployment
Unlike Catalyst 9000 switches that use AppGigabitEthernet interfaces, Cisco routers use VirtualPortGroup (VPG) interfaces for application hosting connectivity. The agent runs as a container inside the IOS-XE Application Hosting framework using KVM or Docker depending on platform and agent version.

Phase 2

Deploy ThousandEyes Enterprise Agents

Install agents on Cisco routers via CLI or SD-WAN Manager.

Pre-Deployment Checklist

  • IOS-XE version meets minimum for your specific model (see Supported Platforms table in Overview)
  • Router has sufficient RAM (8 GB minimum) and storage for application hosting
  • Outbound TCP 443 to *.thousandeyes.com confirmed
  • DNS resolution and NTP synchronization verified on the router
  • Account group token obtained from ThousandEyes portal
  • VirtualPortGroup IP addressing planned (for CLI deployment)
  • For SD-WAN: ThousandEyes VPN configured in SD-WAN Manager

Ideal for standalone router deployments or environments without SD-WAN Manager.

Step 1: Enable IOx Application Hosting Framework

configure terminal
iox
end

Wait approximately 60 seconds, then verify IOx is running:

show iox-service

All IOx services should show as Running.

Step 2: Configure VirtualPortGroup Interface

The VirtualPortGroup (VPG) interface provides network connectivity between IOS-XE and the hosted application container:

configure terminal
!
interface VirtualPortGroup0
 ip address 192.168.2.1 255.255.255.252
 no shutdown
!
end
🔎
VPG Addressing
Use a /30 subnet for the VPG link. The router gets .1, the agent container gets .2. The agent's default gateway points to the VPG router address. The router handles routing for the agent — NAT is not required on the VPG interface for the agent to reach the internet. Only configure NAT if your specific topology requires it (e.g., the VPG subnet is not included in routing tables upstream).

Step 3: Configure Application Hosting

Replace placeholder values with the customer's environment details:

configure terminal
app-hosting appid thousandeyes_agent
 app-vnic gateway0 virtualportgroup 0 guest-interface 0
  guest-ipaddress 192.168.2.2 netmask 255.255.255.252
 exit
 app-default-gateway 192.168.2.1 guest-interface 0
 name-server0 8.8.8.8
 name-server1 8.8.4.4
 app-resource docker
  prepend-pkg-opts
  run-opts 1 "-e TEAGENT_ACCOUNT_TOKEN=<your-token-here>"
  run-opts 2 "--hostname Router-Branch01-Agent"
 exit
end

Step 4: Download and Install the Agent Image

Install directly via HTTPS (if the router has internet access):

app-hosting install appid thousandeyes_agent package https://downloads.thousandeyes.com/enterprise-agent/thousandeyes-enterprise-agent-4.0.cat9k.tar

Or copy to flash first, then install locally:

copy https://downloads.thousandeyes.com/enterprise-agent/thousandeyes-enterprise-agent-4.0.cat9k.tar flash:
app-hosting install appid thousandeyes_agent package flash:thousandeyes-enterprise-agent-4.0.cat9k.tar

Step 5: Activate and Start the Agent

app-hosting activate appid thousandeyes_agent
app-hosting start appid thousandeyes_agent

Step 6: Verify the Agent

show app-hosting list

The output should show the agent in RUNNING state. For detailed status:

show app-hosting detail appid thousandeyes_agent

The agent should appear in the ThousandEyes portal under Cloud & Enterprise Agents > Agent Settings within 2-5 minutes.

🔎
Troubleshooting
If the agent doesn't appear: connect to the container shell with app-hosting connect appid thousandeyes_agent session, then check /var/log/agent/te-agent.log. Common issues include: incorrect token, blocked outbound TCP 443, DNS resolution failure, or insufficient router resources.

💡
SE Tip: Bulk Deployment
SD-WAN Manager allows you to deploy ThousandEyes agents to hundreds of routers simultaneously using a single template push. Use device-specific variables for hostname and IP to ensure unique agent identities. This is the recommended approach for large-scale SD-WAN fabrics.
⚠
VPN Selection Matters
Choosing VPN 0 (transport) gives the agent direct internet access for testing external SaaS/cloud targets. Choosing VPN 1 (service side) places the agent in the enterprise network, ideal for testing internal applications and data center connectivity. Consider deploying agents in both VPNs for full coverage at critical sites.
Phase 3

Traffic Forwarder for Traffic Insights

Configure the router and ThousandEyes agent as a traffic forwarder to gain deep visibility into real user traffic flows.

ThousandEyes Traffic Insights analyzes actual network traffic (NetFlow/IPFIX) to provide visibility into traffic patterns, top talkers, and application usage. By configuring the Cisco router as a traffic forwarder, you export flow data to ThousandEyes for analysis — complementing the active synthetic tests with passive, real-traffic intelligence.

How Traffic Insights Works

  • The router collects NetFlow v9 or IPFIX data from WAN-facing interfaces
  • Flow records are exported to the ThousandEyes Enterprise Agent running on the router (or a nearby collector)
  • The agent processes and forwards summarized flow telemetry to the ThousandEyes cloud
  • ThousandEyes presents traffic analytics in the Traffic Insights dashboard

Option A: Configure via SD-WAN Manager

For SD-WAN deployments, Traffic Insights is configured through SD-WAN Manager using traffic monitoring policies.

1

In SD-WAN Manager, navigate to Configuration > Policies > Traffic Policy (or Configuration > Feature Profiles > Traffic Monitoring in 20.12+).

2

Create or edit a Traffic Monitoring policy and enable ThousandEyes Traffic Monitor.

3

Select the VPN(s) to monitor (typically VPN 0 for transport-side traffic or VPN 1 for service-side).

4

The traffic monitor automatically targets the local ThousandEyes Enterprise Agent as the flow collector.

5

Apply the policy to the target device template and push to routers.

🔎
SD-WAN Traffic Monitor
In SD-WAN mode, the traffic monitor leverages the cflowd (NetFlow) infrastructure already built into the SD-WAN data plane. No additional flow exporter configuration is needed on the router — SD-WAN Manager handles the full configuration via policy.

Option B: Configure via CLI (Non-SD-WAN)

For standalone IOS-XE routers, configure NetFlow export manually to the local ThousandEyes agent.

Step 1: Configure the Flow Exporter

configure terminal
!
flow exporter TE-EXPORTER
 destination 192.168.2.2
 source VirtualPortGroup0
 transport udp 2055
 export-protocol netflow-v9
 template data timeout 60
!
end

The destination IP (192.168.2.2) is the ThousandEyes agent's IP on the VPG interface configured during agent deployment.

Step 2: Configure the Flow Monitor

configure terminal
!
flow record TE-RECORD
 match ipv4 source address
 match ipv4 destination address
 match ipv4 protocol
 match transport source-port
 match transport destination-port
 match interface input
 collect counter bytes long
 collect counter packets long
 collect timestamp sys-uptime first
 collect timestamp sys-uptime last
 collect interface output
!
flow monitor TE-MONITOR
 record TE-RECORD
 exporter TE-EXPORTER
 cache timeout active 60
!
end

Step 3: Apply the Flow Monitor to WAN Interfaces

configure terminal
!
interface GigabitEthernet0/0/0
 description WAN Interface
 ip flow monitor TE-MONITOR input
 ip flow monitor TE-MONITOR output
!
end
write memory

Step 4: Verify Flow Export

show flow monitor TE-MONITOR statistics
show flow exporter TE-EXPORTER statistics

Confirm that flow records are being collected and exported. The ThousandEyes agent will begin processing flow data and forwarding it to the ThousandEyes cloud.

Viewing Traffic Insights in ThousandEyes

1

In the ThousandEyes portal, navigate to Traffic Insights from the left navigation.

2

Select the agent/site to view traffic analytics including top talkers, application distribution, and traffic volume trends.

3

Use filters to drill into specific time ranges, source/destination pairs, or application categories.

💡
SE Tip: Traffic Insights Value
Traffic Insights is a powerful complement to synthetic tests. While tests tell you "can my users reach this service?", Traffic Insights tells you "what are my users actually accessing and how much bandwidth are they consuming?" Together they provide full-stack WAN observability.

Phase 5

SD-WAN Overlay & Underlay Monitoring

Configure ThousandEyes to monitor both the SD-WAN overlay tunnels and the underlying transport network.

Cisco SD-WAN operates on two layers: the overlay (IPsec/GRE tunnels between WAN Edge routers) and the underlay (the physical transport — MPLS, broadband, LTE). ThousandEyes provides visibility into both, enabling rapid fault isolation between provider issues and overlay configuration problems.

Understanding Overlay vs. Underlay

LayerWhat It IsThousandEyes Visibility
OverlaySD-WAN IPsec/GRE tunnels between WAN Edge routersAgent-to-Agent tests between Enterprise Agents on routers for end-to-end tunnel performance measurement
UnderlayPhysical transport (MPLS, broadband, 4G/5G) between sitesAgent-to-Server tests targeting ISP gateways, MPLS PE routers, or internet endpoints; Path Visualization shows per-hop underlay performance

Monitoring the SD-WAN Overlay

The overlay is the encrypted tunnel fabric that SD-WAN builds between sites. Monitoring it reveals tunnel-level performance and helps detect issues like tunnel flaps, asymmetric routing, or overlay congestion.

Method 1: Agent-to-Agent Tests (Active Probing)

Create Agent-to-Agent tests between Enterprise Agents deployed on WAN Edge routers at different sites. This measures the end-to-end overlay performance as experienced by the SD-WAN fabric.

SettingValueNotes
Test TypeAgent to AgentNetwork category
Source AgentBranch router agente.g., Branch-NYC-Agent
Target AgentHub/DC router agente.g., DC-Dallas-Agent
ProtocolTCP (recommended)Traverses SD-WAN overlay like real traffic
Port49153 (default) or customEnsure port is reachable across overlay
DirectionBoth directionsMeasures bidirectional performance
Interval2 or 5 minutesLower interval for critical hub-spoke paths
ThroughputEnable (optional)Measures available bandwidth across overlay

Monitoring the SD-WAN Underlay

The underlay is the physical transport that carries the encrypted SD-WAN tunnels. Monitoring it helps identify whether performance issues originate from the ISP/carrier, the local last mile, or the SD-WAN overlay itself.

Method 1: Agent-to-Server Tests to ISP/Transport Targets

Create Agent-to-Server tests targeting known underlay endpoints to measure raw transport performance independent of the SD-WAN overlay.

Target TypeExample TargetWhat It Reveals
ISP Gateway / Next HopISP PE router IPLast-mile and ISP backbone performance
MPLS PE RouterProvider PE loopback IPMPLS transport quality
Public Internet Endpoint8.8.8.8 or well-known SaaS IPEnd-to-end internet path quality
Regional Cloud GatewayAWS/Azure regional endpointCloud on-ramp performance

Configure each as an Agent-to-Server test with TCP protocol and 5-minute intervals. Enable Path Visualization to see every hop in the underlay path.

Method 2: Path Visualization Analysis

ThousandEyes Path Visualization on any test automatically maps the underlay path, showing:

  • Every router hop between the branch agent and the target
  • Per-hop latency and loss to pinpoint degradation at specific network segments
  • MPLS labels revealing the provider's internal routing decisions
  • ASN transitions showing where traffic crosses between networks
  • Link-layer changes that indicate rerouting or failover events
💡
SE Tip: Overlay + Underlay Correlation
The real power is correlating overlay and underlay data. When an Agent-to-Agent test (overlay) shows degradation, check the Agent-to-Server underlay tests for the same timeframe. If the underlay is also degraded, the issue is with the transport provider. If the underlay is clean but the overlay is impacted, investigate SD-WAN policy, tunnel configuration, or encryption overhead.

Phase 6

Validate and Next Steps

Confirm everything is working and plan for expanded monitoring.

Validation Checklist

  • Enterprise Agent(s) appear in ThousandEyes portal under Cloud & Enterprise Agents > Agent Settings
  • Agent status shows "Online" with green indicator
  • Agent-to-Server test to data center shows data with path visualization
  • HTTP Server test(s) to SaaS applications show availability and timing data
  • DNS Server test shows successful resolution with expected IPs
  • Traffic Insights shows flow data from the router (if configured)
  • SD-WAN overlay tests (Agent-to-Agent) show bidirectional metrics
  • SD-WAN underlay tests show path visualization with per-hop data
  • Alert rules are active and notification channels configured

Recommended Next Steps

  • Walk the customer through dashboard views (Overview, Path Visualization, Table view)
  • Create a custom dashboard with widgets for WAN health, SaaS performance, and DNS status
  • Configure alert notifications (email, Slack, PagerDuty, ServiceNow webhooks)
  • Set up Snapshot Sharing so the customer can share test results with ISPs for troubleshooting
  • Explore ThousandEyes + Catalyst Center / SD-WAN Manager embedded views for unified operations
  • Deploy agents on additional routers at other branch sites for broader coverage
  • Add Web Transaction tests for multi-step application workflows (e.g., login flows)
  • Enable Internet Insights for macro-level outage detection across ISPs and cloud providers
